Operating Systems

Topic 2: Process, Process API & Limited Direct Execution

What is a Process?

One of the most fundamental abstractions an OS provides is the process — simply defined as a running program. A program sitting on disk is just bytes; when the OS loads it and begins executing its instructions, it becomes a process.

To characterize a process fully, we identify everything the system accesses or affects during its execution:

Memory (Address Space)
  • Instructions (text/code)
  • Data section (globals & statics)
  • Heap (dynamic allocations)
  • Stack (local variables, return addresses)
CPU State (Registers)
  • Program Counter (PC / Instruction Pointer)
  • Stack Pointer
  • General-purpose registers
  • I/O information (open file descriptors)
CPU Virtualization: The OS creates the illusion that many virtual CPUs exist by time sharing — running one process for a short while, stopping it, and running another. The potential cost is performance overhead.

Process Address Space

Every process is given its own private virtual address space, isolated from all other processes. This guarantees stability and security. The address space is organized into well-defined segments:

Text & Data

Text: executable code of the program.
Data: global and static variables.

Heap

Dynamic memory allocation. Grows upward. Managed with malloc() / free().

Stack

Function call frames, local variables, return addresses. Grows downward.

A fourth segment, the memory-mapped segment, stores shared libraries, dynamically loaded modules, and memory-mapped files.

Process Creation: From Program to Process

When the OS creates a new process it follows a specific sequence of steps:

1

Load
Copy code & data from disk into memory (lazily)

2

Stack
Allocate run-time stack; initialize with argv/argc

3

Heap
Create the heap for dynamic allocations

4

I/O
Open stdin, stdout, stderr (file descriptors 0, 1, 2)

5

Start
Transfer control to main()

Lazy loading: The OS does not load the entire program into memory before starting it. Instead it loads pieces of code and data only as they are needed during execution, keeping startup fast and memory usage low.

Process States

At any instant a process is in one of three states:

Running

The process is executing instructions on a processor right now.

Ready

The process is ready to run but the OS has chosen not to run it at this moment.

Blocked

The process is waiting for an event (e.g. I/O completion) and cannot run until that event occurs.

The transitions between states follow fixed rules:

  • Running → Ready: descheduled (time-slice expiration or higher-priority process arrives)
  • Ready → Running: scheduled by the OS
  • Running → Blocked: process initiates I/O (or other wait)
  • Blocked → Ready: I/O completes

Process Control Block (PCB) & Data Structures

The OS is itself a program and it tracks every process using a data structure called the Process Control Block (PCB). The OS maintains a process list that holds PCBs for all ready, blocked, and currently running processes.

A typical PCB contains:

Identity & State
  • Process ID (PID)
  • Process state (running / ready / blocked)
  • Parent/child process references
CPU & Memory Context
  • Program Counter (PC)
  • Saved register values
  • Memory info (base/limit, page tables)
  • I/O info (open files, devices)
  • CPU scheduling info (priority, time quantum)

The xv6 kernel uses a struct proc as its PCB and a separate struct context to save the registers needed to resume a process:

struct context {
    int eip;  // Index pointer register
    int esp;  // Stack pointer register
    int ebx;  // Base register
    int ecx;  // Counter register
    int edx;  // Data register
    int esi;  // Source index register
    int edi;  // Destination index register
    int ebp;  // Stack base pointer register
};

Process states in xv6: UNUSED, EMBRYO, SLEEPING, RUNNABLE, RUNNING, ZOMBIE.

Process API: fork(), wait(), exec()

fork() — Create a Child Process

fork() creates an almost exact copy of the calling process. After fork() returns, there are two processes running, both about to return from the call. The child gets its own separate memory space (same initial contents), its own registers and PC.

Return value rule: fork() returns the child's PID to the parent, and returns 0 to the child. A negative return means fork failed. This return-value difference is the standard way to distinguish parent from child code paths. 
int rc = fork();
if (rc < 0) {
    fprintf(stderr, "fork failed\n"); exit(1);
} else if (rc == 0) {
    // child process
    printf("hello, I am child (pid:%d)\n", (int) getpid());
} else {
    // parent process
    printf("hello, I am parent of %d (pid:%d)\n", rc, (int) getpid());
}
Non-deterministic ordering: The CPU scheduler decides which process (parent or child) runs first after fork(). Output order is not guaranteed without explicit synchronization.

wait() — Synchronize with a Child

wait() makes the parent block until one of its child processes exits. This enforces ordering: if the parent calls wait(), the child always finishes and prints first, making execution deterministic.

int wc = wait(NULL);
printf("hello, I am parent of %d (wc:%d) (pid:%d)\n",
       rc, wc, (int) getpid());

exec() — Replace Process Image

exec() (e.g. execvp()) replaces the current process's memory with a completely new program. It takes the binary name and an argument array. Crucially, exec() does not return on success — the process is now running the new program.

char *myargs[3];
myargs[0] = strdup("wc");    // program to run
myargs[1] = strdup("p3.c"); // argument
myargs[2] = NULL;            // end of args
execvp(myargs[0], myargs);  // replaces this process with "wc"
printf("this should NOT print");
Why separate fork() and exec()? The gap between the two calls lets the shell manipulate the child's environment before the new program starts — for example, redirecting I/O or setting up pipes.

I/O Redirection and Pipes

Shell commands like wc p3.c > out.txt work by: (1) fork()ing a child, (2) closing STDOUT_FILENO in the child, (3) opening the output file (which gets file descriptor 1 because it is the lowest available), and then (4) calling exec("wc", ...). The wc program writes to stdout not knowing it has been redirected.

Pipes (|) connect the STDOUT of one process to the STDIN of another. Advantages over redirection with temporary files: automatic cleanup, no disk space required, and reader/writer proceed in parallel.

Limited Direct Execution

The OS wants to run programs as fast as possible (direct execution), but must remain in control. This tension defines the Limited Direct Execution (LDE) protocol.

Challenge 1: Restricted Operations

A user process must not be able to directly access all hardware resources. How do we allow useful operations while preventing dangerous ones?

Challenge 2: Switching Between Processes

Once a process is running, how does the OS regain control of the CPU so it can switch to another process?

Solution to Challenge 1: Processor Modes & System Calls

Hardware provides two privilege levels:

User Mode

Applications run here. Restricted access to hardware. Cannot directly issue I/O, modify OS data structures, or change privilege level.

Kernel Mode

OS runs here. Full access to all hardware resources and privileged instructions.

When a user process needs a privileged operation it executes a trap instruction (int n). This raises the privilege level to kernel mode, saves registers on the kernel stack, and jumps to the OS's trap handler. After handling, a return-from-trap instruction restores registers, reduces privilege back to user mode, and returns to the caller.

Trap table: At boot time, the OS tells the hardware the address of every trap handler. The hardware stores this trap table. When a trap occurs, hardware looks up the correct handler by trap number. User code cannot change the trap table — only kernel-mode code can.

Sources of traps include: completion of disk I/O, keyboard interrupts, and system calls (e.g. open(), fork(), write()).

Solution to Challenge 2: Timer Interrupt

A cooperative approach relies on processes voluntarily giving up the CPU via system calls. Problem: a process stuck in an infinite loop never yields.

The non-cooperative (timer interrupt) approach solves this: during boot the OS programs a hardware timer to raise an interrupt every few milliseconds. When the interrupt fires:

  1. The currently running process is suspended.
  2. Enough state (registers, PC) is saved.
  3. A pre-configured interrupt handler in the OS runs.
Key insight: A timer interrupt gives the OS the ability to regain control of the CPU periodically — even if the running process never cooperates. This is what makes preemptive multitasking possible.

Context Switch

After the timer interrupt, the OS scheduler decides whether to continue the current process or switch to another. If it switches, a context switch occurs:

  1. Save current process's registers (general-purpose, PC, kernel stack pointer) onto its kernel stack.
  2. Restore the next process's registers from its kernel stack.
  3. Switch the stack pointer to the new process's kernel stack.

The xv6 swtch() function implements this in assembly, explicitly saving and restoring all eight registers (eip, esp, ebx, ecx, edx, esi, edi, ebp).

Concurrency during interrupt handling: What if another interrupt arrives while the OS is handling one? The OS disables interrupts during interrupt processing, or uses locking schemes to protect internal data structures from concurrent modification.
