Topic 3: Scheduling, MLFQ, Address Spaces & Address Translation

Scheduling Metrics & Basic Algorithms

A scheduler must decide which ready process gets the CPU next. To compare schedulers fairly, we use two core metrics:

Turnaround Time

Tturnaround = Tcompletion − Tarrival
Measures how long a job takes from arrival to completion. Optimizing turnaround favors running shorter jobs first.

Response Time

Tresponse = Tfirstrun − Tarrival
Measures how quickly a job first gets the CPU. Critical for interactive systems where users expect immediate feedback.

The core trade-off: Optimizing turnaround time typically hurts response time and vice versa. Optimizing fairness hurts turnaround. No single algorithm wins on all metrics simultaneously.

First In, First Out (FIFO / FCFS)

Jobs run in arrival order to completion. Simple and easy to implement, but suffers from the convoy effect — short jobs get stuck behind a long-running job, causing poor average turnaround time. Example: A(100s), B(10s), C(10s) all arrive at t=0; average turnaround = (100+110+120)/3 = 110s.

Shortest Job First (SJF)

Run the shortest job first, then the next shortest. Non-preemptive. Optimal turnaround when all jobs arrive simultaneously. Fails badly when jobs arrive at different times — a long job that started early cannot be preempted even if a shorter job arrives.

Shortest Time-to-Completion First (STCF)

Adds preemption to SJF. When a new job arrives, compare its remaining time to all other jobs' remaining times and run whichever has the least time left. Also called Preemptive Shortest Job First (PSJF). Optimal turnaround time among preemptive schedulers.

STCF example: A arrives at t=0 (needs 100s); B and C arrive at t=10 (each needs 10s). STCF preempts A, runs B then C, then resumes A. Average turnaround = (120 + (20−10) + (30−10)) / 3 = 50s — far better than SJF's 103.33s in this scenario.

Round Robin (RR)

Time-slicing scheduler: run each job for a fixed time slice (quantum), then switch to the next job in the queue, cycling until all jobs finish. Excellent for response time but poor for turnaround time. The length of the time slice is a critical parameter:

Short time slice

Better response time. But context-switching overhead dominates and throughput drops.

Long time slice

Lower context-switch overhead. But response time worsens and the system becomes less interactive.

Incorporating I/O

When a job issues an I/O request it blocks and no longer needs the CPU. The scheduler should overlap computation with I/O: while one job waits for I/O to complete, another job runs on the CPU. When the I/O finishes, the OS raises an interrupt and moves the job back to the ready state.

Multi-Level Feedback Queue (MLFQ)

MLFQ solves a fundamental dilemma: how to optimize turnaround time (like SJF) without knowing job lengths in advance, while also minimizing response time for interactive jobs. It learns from past behavior to predict the future.

Basic Structure

MLFQ maintains several queues, each with a different priority level. A job ready to run sits in exactly one queue. Jobs in higher-priority queues run before lower-priority ones. Among jobs in the same queue, Round Robin is used.

Rule 1: If Priority(A) > Priority(B), A runs (B doesn't).
Rule 2: If Priority(A) = Priority(B), A & B run in Round Robin.
Rule 3: When a job enters the system, it is placed at the highest priority.
Rule 4: Once a job uses up its time allotment at a given level (regardless of how many times it has given up the CPU), its priority is reduced.
Rule 5: After some time period S, move all jobs in the system to the topmost queue (Priority Boost).

Priority Adjustment Logic

MLFQ infers job type from behavior. A job that repeatedly gives up the CPU (I/O-bound, interactive) stays at high priority. A job that keeps consuming its full time slice (CPU-bound, long-running) gets demoted to a lower queue. This approximates SJF without needing to know job lengths upfront.

Problems with Naive MLFQ

Starvation

Too many interactive jobs at high priority can starve long-running CPU jobs, which never receive CPU time.

Gaming

A malicious job can issue an I/O just before its time slice expires to stay at high priority and monopolize the CPU.

Behavior Change

A job that starts CPU-bound may later become I/O-bound. Without priority boost, it remains stuck at low priority forever.

Solutions

Priority Boost (Rule 5): Periodically (every period S), move all jobs to the topmost queue. This prevents starvation and adapts to behavior changes. Better Accounting (Rule 4 rewrite): Track the total CPU time a job uses at a level — not just whether it voluntarily yielded. Once it exhausts its allotment, demote it, regardless of how many I/O yields it performed. This defeats gaming.

MLFQ Tuning in Real Systems

Higher-priority queues use shorter time slices (e.g., 10ms) for snappy response. Lower-priority queues use longer time slices (e.g., 100ms) to amortize context-switch cost for CPU-bound jobs. Solaris implements MLFQ with 60 queues, slowly increasing time slices, and a priority boost roughly every 1 second. FreeBSD computes priority from CPU usage history (decay-based formula) rather than discrete queues.

Address Spaces — Memory Virtualization

Early systems loaded one program into memory at a time — the OS was simply a library sitting at the bottom of RAM. As machines became more expensive, multiprogramming emerged: load multiple processes into memory and switch between them to maximize CPU utilization.

The Address Space Abstraction

The OS creates an address space — a clean abstraction that gives each process the illusion of owning all of memory. Every address space contains three key regions:

Code (Text)

Where program instructions live. Loaded from the executable. Typically read-only and placed at low addresses (e.g., 0KB).

Heap

Dynamically allocated memory (malloc in C, new in OO languages). Grows upward from after the code segment.

Stack

Stores return addresses, local variables, and function arguments. Grows downward from the top of the address space.

Every address a running program sees is virtual. The OS (with hardware help) translates each virtual address into the actual physical location in RAM. Programs never see or touch physical addresses directly. 
#include <stdio.h>
#include <stdlib.h>
int main() {
    printf("code  : %p\n", (void *) main);
    printf("heap  : %p\n", (void *) malloc(1));
    int x = 3;
    printf("stack : %p\n", (void *) &x);
    return 0;
}

On a 64-bit Linux machine this prints three very different virtual addresses: code near 0x400000, heap near 0xcf2000, and stack near 0x7fff9ca28000 — all virtual, all translated to physical RAM by the OS at runtime.

Goals of Memory Virtualization

  • Ease of use: Each process sees a simple, large, contiguous address space starting at 0.
  • Efficiency: Translation must be fast and use minimal memory overhead.
  • Isolation & protection: One process cannot read or write another process's memory, nor corrupt the OS.

Address Translation — Base and Bounds

The OS virtualizes memory using a strategy similar to Limited Direct Execution for the CPU: let the hardware do the translation efficiently, but keep the OS in control of setting things up and handling exceptions.

Dynamic Relocation

The simplest hardware mechanism is the base and bounds register pair (one pair per CPU). When the OS loads a process, it sets:

Base Register

Holds the starting physical address of the process's address space. Every virtual address is added to this value to get the physical address.

Bounds Register

Holds the size (or end) of the address space. Any access beyond this limit raises a hardware exception — protecting other processes and the OS.

Translation formula: physical address = virtual address + base
Bounds check: 0 ≤ virtual address < bounds (else: protection fault) 
; Example: base = 32KB, process virtual address space = 16KB
128 : movl 0x0(%ebx), %eax   ; virtual fetch at 128  → physical 32KB + 128 = 32896
132 : addl $0x03, %eax        ; no memory reference
135 : movl %eax, 0x0(%ebx)   ; virtual store at 15KB → physical 32KB + 15KB = 47KB

Hardware Requirements for Base & Bounds

  • Privileged mode: User processes cannot execute privileged instructions (e.g., modifying base/bounds registers).
  • Base and bounds registers: One pair per CPU for address translation and bounds checking.
  • Translation circuitry: Hardware computes physical address and checks bounds on every memory access.
  • Exception handling: Hardware raises an exception on out-of-bounds access; OS registers exception handlers.

OS Responsibilities at Three Junctures

Process Start

OS searches the free list for a contiguous region of physical memory large enough for the process's address space, then sets base and bounds.

Process Termination

OS places the memory region back on the free list so it can be reused by future processes.

Context Switch

OS saves the current base/bounds values into the Process Control Block (PCB) and loads the new process's saved values into the hardware registers.
