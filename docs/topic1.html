Operating Systems

Topic 1: Introduction — What is an OS and what does it do?

What Happens When a Program Runs?

At its most basic level, a running program simply executes instructions. This follows the Von Neumann model of computing — a cycle that every modern processor repeats billions of times per second:

1

Fetch
Processor retrieves an instruction from memory

2

Decode
Figure out which instruction this is

3

Execute
Do what the instruction says (add, branch, load, store…)

4

Advance
Move to the next instruction and repeat

The processor repeats this fetch-decode-execute loop until the program completes. But who manages all the programs running on the system, allocates memory to them, and lets them share the hardware? That is the job of the Operating System.

What is an Operating System?

The Operating System (OS) is a layer of software that sits between application programs and the hardware. Its three core responsibilities are:

  • Making it easy to run programs
  • Allowing programs to share memory
  • Enabling programs to interact with devices
Key definition: The OS is in charge of making sure the system operates correctly, efficiently, and is easy to use.

The OS can be thought of in three complementary ways:

Virtual Machine

Takes physical resources and presents a more powerful, general, easy-to-use virtual form of them.

Standard Library

Exports a few hundred system calls (APIs) that programs use to run code, access memory, and use devices.

Resource Manager

Manages CPU, memory, and disk — sharing them efficiently among many concurrently running programs.

Three Major Themes of the Course

Virtualization

"You own the whole machine." The OS creates an illusion that each program has exclusive access to CPU, memory, and disk — even when many programs share them.

Concurrency

"Many things are happening at once." Managing multiple threads and processes that execute simultaneously introduces subtle correctness problems.

Persistence

"Your data survives crashes." The file system and storage hardware ensure data is not lost when power fails or the system crashes.

Virtualizing the CPU

Even on a machine with a single physical processor, the OS creates the illusion of many virtual CPUs. Consider this simple C program cpu.c:

int main(int argc, char *argv[]) {
    char *str = argv[1];
    while (1) {
        Spin(1);          // busy-wait for 1 second
        printf("%s\n", str);
    }
    return 0;
}

When four instances are launched simultaneously, the output interleaves A, B, C, D — even though only one physical CPU exists. The OS achieves this by rapidly time-sharing the CPU among processes.

CPU Virtualization: Turning a single CPU into a seemingly infinite number of CPUs, allowing many programs to appear to run at the same time. The OS (with hardware help) is responsible for this illusion.

Two important questions arise from CPU virtualization:

How do programs communicate with the OS?

Answer: API (System Calls) — programs use the OS-provided interface to run programs, stop them, or tell the OS which program to run.

Which program runs when multiple want to run?

Answer: Policy — the OS uses scheduling policies to decide which process gets the CPU at any given moment.

Virtualizing Memory

Physical memory is simply an array of bytes. Every program reads memory (load) by specifying an address, and writes memory (store) by specifying an address and data. The OS creates a virtual address space for each process.

int *p = malloc(sizeof(int));  // allocate memory
*p = 0;                         // initialize
while (1) {
    Spin(1);
    *p = *p + 1;
    printf("(%d) p: %d\n", getpid(), *p);
}

When two copies of this program run simultaneously, both print address 00200000, yet they count independently. This reveals a key property:

Virtual Address Space: Each process has its own private virtual address space. The OS maps each virtual address onto physical memory. A memory reference in one process does not affect another process's memory — even if the virtual addresses are the same. Physical memory is a shared resource managed entirely by the OS.

A Process Identifier (PID) is a unique number assigned by the OS to each running process so it can manage and track processes efficiently. A process is an instance of a program in execution.

The Problem of Concurrency

Both the OS (juggling many processes) and modern multi-threaded programs exhibit concurrency. Consider thread.c — two threads both increment a shared counter:

volatile int counter = 0;

void *worker(void *arg) {
    for (int i = 0; i < loops; i++) {
        counter++;      // this is NOT atomic!
    }
    return NULL;
}
// main creates two worker threads, then joins them

With loops = 1000 the answer is reliably 2000. With loops = 100000 the result is wrong and non-deterministic (e.g., 143012 one run, 137298 the next).

Why does this happen?

The statement counter++ compiles to three separate machine instructions:

  1. Load the value of counter from memory into a register
  2. Increment the register value
  3. Store the register value back into memory
Non-atomic operations cause race conditions. If a thread is interrupted between the load and the store — and another thread executes — both threads may read the same old value, increment it, and write back the same result, losing one increment. This is the fundamental problem of concurrency.

Persistence

Volatile memory (DRAM) loses all data when power is removed or the system crashes. Persistent storage requires both hardware and software working together:

Hardware
  • Hard Disk Drives (HDDs)
  • Solid-State Drives (SSDs)
  • Any non-volatile I/O device
Software (File System)
  • Manages the disk on behalf of the user
  • Stores files created by user programs
  • Handles crash recovery (journaling, copy-on-write)

Writing a file involves system calls routed to the file system:

int fd = open("/tmp/file", O_WRONLY | O_CREAT | O_TRUNC, S_IRWXU);
int rc = write(fd, "hello world\n", 13);
close(fd);

The OS file system must figure out where on disk the data will reside, issue the necessary I/O requests to the storage device, and carefully order writes to survive crashes — using techniques such as journaling or copy-on-write.

OS Design Goals

Building a good OS requires balancing several competing design goals:

Abstraction

Hide hardware complexity. Make the system convenient and easy to use by providing clean, high-level interfaces.

High Performance

Minimize OS overhead. Virtualization must not impose excessive cost on programs. Every extra instruction the OS executes is overhead.

Protection & Isolation

Bad behavior of one application must not harm other applications or the OS itself. Each process is isolated from others.

Reliability

The OS must run non-stop. Additional considerations include energy-efficiency, security, and mobility for modern platforms.

What Resources Does the OS Manage?

The OS deals with three fundamental hardware resources in every system:

  • CPU — executes code with data (virtualized as processes/threads)
  • Memory — stores code and data (virtualized as address spaces)
  • Storage — persistently stores code and data (managed by the file system)

From mainframe computers (M360) running early virtual machines, to Unix on PDP-11 minicomputers introducing shell, pipes, and signals, to personal computers running DOS/Windows/Mac, to today's smartphones, IoT devices, self-driving cars, and AI systems — an operating system lies behind all of them.
