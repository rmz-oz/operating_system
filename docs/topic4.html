Topic 4: Segmentation & Paging — Memory Virtualization Techniques

The Problem with Base and Bounds

In the base-and-bounds approach, each process gets a single contiguous region in physical memory defined by a base register and a size (bounds) register. While simple, this has a critical weakness: the entire address space — including the large chunk of free space between the heap and stack — must fit in physical memory.

Key inefficiency: The free space between heap and stack is mapped into physical memory even though no data lives there. This wastes physical RAM. Additionally, if the entire address space does not fit in physical memory, the program cannot run at all.
Address Space Layout
  • 0 KB — Code (used)
  • 2 KB — Heap (used)
  • 4–14 KB — Free space (wasted in physical RAM)
  • 14 KB — Stack (used)
Problems
  • Free space occupies physical memory
  • Hard to run programs larger than physical RAM
  • Only one base/bounds pair for the whole space

Segmentation

Segmentation solves the base-and-bounds inefficiency by giving each logically distinct region of the address space its own base and bounds pair. Instead of one pair for the whole process, there is one pair per segment.

Definition: A segment is a contiguous portion of the address space of a particular length. The three classic segments are Code, Heap, and Stack. Each can be placed anywhere in physical memory independently.

With segmentation, the segment table might look like this for a process:

Segment   Base    Size
Code      32 KB   2 KB
Heap      34 KB   2 KB
Stack     28 KB   2 KB

The OS can now place each segment in any available region of physical memory. The large "free" space between heap and stack in the virtual address space does not need any physical memory backing.

Address Translation with Segmentation

The key formula for translation is:

physical address = offset + base

The offset is how far the referenced address is from the start of that segment. For example, virtual address 100 falls in the code segment (which starts at virtual address 0), so the offset is 100. With a base of 32 KB:

physical address = 100 + 32768 = 32868

For the heap, the calculation is slightly different. Virtual address 4200 falls in the heap (which starts at virtual address 4096), so the offset is 4200 - 4096 = 104. With the heap base at 34 KB:

physical address = 104 + 34816 = 34920
Common mistake: Do NOT add the raw virtual address to the base. You must first compute the offset within the segment (virtual address minus the segment's starting virtual address), then add the base.

Segmentation Fault

If a program accesses an address that lies beyond the end of any valid segment (for example, virtual address 7 KB when the heap ends at 6 KB), the hardware detects an out-of-bounds access. The OS raises a segmentation fault and the process is terminated.

Referring to Segments — The Explicit Approach

When the hardware receives a virtual address, it must determine which segment the address belongs to. In the explicit approach, the top few bits of the virtual address encode the segment number.

For a 14-bit virtual address space with 3 segments, the top 2 bits indicate the segment:

Bits 13–12 = 00

Code segment

Bits 13–12 = 01

Heap segment

Bits 13–12 = 10

Stack segment

Example: virtual address 4200 in binary is 01000001101000. Bits 13–12 are 01 → Heap. The remaining 12 bits give the offset within the heap. The hardware uses:

Segment = (VirtualAddress & SEG_MASK) >> SEG_SHIFT
Offset  = VirtualAddress & OFFSET_MASK
if (Offset >= Bounds[Segment])
    RaiseException(PROTECTION_FAULT)
else
    PhysAddr = Base[Segment] + Offset
    Register = AccessMemory(PhysAddr)

The constants used are: SEG_MASK = 0x3000 (top 2 bits), SEG_SHIFT = 12, OFFSET_MASK = 0xFFF (bottom 12 bits).

Stack and Sharing Support in Segmentation

Stack Grows Backwards

The stack grows in the negative direction (toward lower addresses). The hardware needs extra support — a Grows Positive bit per segment — to handle this correctly.

Grows Positive = 1 (Code, Heap)

Offset is measured forward from the base. Normal translation applies.

Grows Positive = 0 (Stack)

Offset is measured backward. The hardware subtracts to find the correct negative offset from the segment base.

Segment Sharing and Protection Bits

Two different address spaces can share the same physical segment (e.g., a read-only code segment shared between two processes running the same program). This requires protection bits per segment:

Segment   Base   Size   Grows Positive?   Protection
Code      32 KB  2 KB   1                 Read-Execute
Heap      34 KB  2 KB   1                 Read-Write
Stack     28 KB  2 KB   0                 Read-Write

Coarse-Grained vs. Fine-Grained Segmentation

Coarse-Grained

Small number of segments — code, heap, stack. Simple, but inflexible. Supported by a few hardware registers.

Fine-Grained

Many segments — greater flexibility for the address space layout. Requires a hardware-supported segment table in memory.

External Fragmentation — The Problem with Segmentation

Over time, as processes start and stop, physical memory develops many small holes of free space between segments. This is called external fragmentation.

Example: There may be 24 KB free total, but split into 3 separate 8 KB holes. A new segment requiring 20 KB of contiguous space cannot be allocated even though 24 KB is nominally free.

One solution is compaction: stop all processes, copy their data to compact segments together, and update all segment registers. Compaction is costly — it involves stopping processes, copying large amounts of data, and rewriting hardware registers. Modern systems prefer paging instead.

Paging — Introduction

Paging takes a different approach: instead of variable-sized segments, it divides both the virtual address space and physical memory into fixed-size units.

Virtual Address Space

Split into fixed-size units called pages.

Physical Memory

Split into fixed-size units called page frames. Pages and page frames are the same size.

Advantages of Paging:
1. Flexibility — no assumption about how heap/stack grow; address space abstraction is easy.
2. Simplicity — all pages are the same size, so free-space management is trivial (just a free list of available frames).

Example: Simple Paging

Consider a 64-byte address space with 16-byte pages (4 pages: 0–3) and a 128-byte physical memory with 16-byte frames (8 frames: 0–7). The OS keeps a page table per process mapping virtual page numbers (VPNs) to physical frame numbers (PFNs):

Virtual Page 0 → Physical Frame 3
Virtual Page 1 → Physical Frame 7
Virtual Page 2 → Physical Frame 5
Virtual Page 3 → Physical Frame 2

The pages can be scattered throughout physical memory in any available frames — there is no requirement for contiguity.

Address Translation in Paging

A virtual address is split into two fields:

VPN — Virtual Page Number

The top bits of the virtual address. Used as an index into the page table to find the physical frame number.

Offset

The bottom bits of the virtual address. Identifies the byte within the page. Copied unchanged into the physical address.

Example: virtual address 21 in a 64-byte address space (6-bit addresses, 4-bit pages, 4-bit offset). In binary: 010101. VPN = 01 = page 1 → frame 7. Offset = 0101 = 5. Physical address = frame 7 start (112) + 5 = 117.

VPN    = (VirtualAddress & VPN_MASK) >> SHIFT
PTEAddr = PTBR + (VPN * sizeof(PTE))   // Page Table Base Register
PTE    = AccessMemory(PTEAddr)
if (PTE.Valid == False)
    RaiseException(SEGMENTATION_FAULT)
else if (CanAccess(PTE.ProtectBits) == False)
    RaiseException(PROTECTION_FAULT)
else
    offset  = VirtualAddress & OFFSET_MASK
    PhysAddr = (PTE.PFN << PFN_SHIFT) | offset
    Register = AccessMemory(PhysAddr)

Where Are Page Tables Stored?

Page tables can be very large. For a 32-bit address space with 4 KB pages:

  • Page offset = 12 bits (log2 of 4096)
  • VPN = 20 bits → 220 = about 1 million entries
  • At 4 bytes per entry: 4 MB per process

Because page tables are too large to fit in hardware registers, they are stored in main memory (usually in kernel-controlled space). The CPU holds a Page Table Base Register (PTBR) that points to the start of the current process's page table in memory.

Performance cost: Every memory access now requires two memory accesses — one to fetch the PTE from the page table, and one to access the actual data. This makes paging slow without additional hardware support (TLB, covered in Topic 5).

Page Table Entry (PTE) Flags

Each entry in the page table stores the Physical Frame Number (PFN) plus several flag bits:

Valid Bit

Indicates whether this translation is valid. If clear and the process accesses this page, a fault is raised.

Protection Bit

Encodes read/write/execute permissions for the page.

Present Bit

Indicates whether the page is currently in physical memory or has been swapped to disk.

Dirty Bit & Reference Bit

Dirty: page has been written since loaded. Reference/Accessed: page has been read. Used by page-replacement algorithms.

An x86 Page Table Entry packs many of these flags into a 32-bit word alongside the PFN. Key x86 bits: P (present), R/W (read/write), U/S (user/supervisor), A (accessed), D (dirty).

Paging is Too Slow — A Memory Trace

Consider a simple loop over an array in C:

int array[1000];
for (i = 0; i < 1000; i++)
    array[i] = 0;

This compiles to 4 assembly instructions that repeat 1000 times:

0x1024  movl $0x0,(%edi,%eax,4)   // store 0 to array[i]
0x1028  incl %eax                  // i++
0x102c  cmpl $0x03e8,%eax          // compare i to 1000
0x1030  jne  0x1024                // loop back if not done

With paging, every single memory reference triggers an extra memory access to fetch the PTE. For the 4 code instructions plus 1 array write per iteration, that is 5 user memory accesses → 5 additional page table lookups per iteration. For 1000 iterations, thousands of extra memory references just for page table walks — a significant overhead.

Solution preview: The Translation Lookaside Buffer (TLB) — a hardware cache for page table entries — is the primary solution to this performance problem (covered in later topics).

Segmentation vs. Paging — Summary Comparison

Segmentation
  • Variable-sized segments
  • Few base/bounds register pairs
  • Supports code, heap, stack naturally
  • Causes external fragmentation
  • Compaction is the fix (costly)
  • Used in early x86 systems
Paging
  • Fixed-size pages (no fragmentation holes)
  • Requires a per-process page table in memory
  • No assumption about heap/stack growth
  • Eliminates external fragmentation
  • Extra memory access per reference (needs TLB)
  • Used in modern x86-64 systems
