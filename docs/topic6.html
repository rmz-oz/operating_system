Operating Systems

Topic 6: Concurrency, Thread API & Locks

1. Concurrency: An Introduction

A thread is a new abstraction for a single running process. A multi-threaded program has more than one point of execution — effectively multiple Program Counters (PCs) that run concurrently. Unlike separate processes, all threads within the same program share the same address space and can therefore access the same data directly.

Each thread has its own PC and its own set of registers. When the OS switches from thread T1 to thread T2, the register state of T1 is saved into a Thread Control Block (TCB) and T2's state is restored — much like a process context switch, but the address space stays the same.

Key difference from processes: In a process context switch the page table (address space) is swapped. In a thread context switch the address space is shared; only registers and the stack pointer change.

Per-Thread Stack

Each thread gets its own stack carved out of the shared address space. This means a two-threaded program has two stacks — Stack(1) at the top of memory and Stack(2) below the free region — while code and heap remain shared.

Why Use Threads?

Parallelism
  • Distribute work across multiple CPUs
  • Transforms slow single-threaded work into parallel computation
  • Called parallelization
Overlap I/O with Computation
  • One thread can block on slow I/O
  • Other threads continue running
  • Similar to how multiprogramming helps across processes

Thread Creation Example

#include <stdio.h>
#include <pthread.h>

void *mythread(void *arg) {
    printf("%s\n", (char *) arg);
    return NULL;
}
int main(int argc, char *argv[]) {
    pthread_t p1, p2;
    printf("main: begin\n");
    Pthread_create(&p1, NULL, mythread, "A");
    Pthread_create(&p2, NULL, mythread, "B");
    Pthread_join(p1, NULL);
    Pthread_join(p2, NULL);
    printf("main: end\n");
    return 0;
}

The execution order of "A" and "B" is non-deterministic — the scheduler decides which thread runs first. Three valid orderings exist: A then B, B then A, or interleaved.

2. Race Conditions and the Critical Section

A race condition occurs when the result of a computation depends on the timing of thread execution. Consider two threads each executing counter = counter + 1. At the assembly level this expands to three instructions:

mov 0x8049a1c, %eax   ; load counter into register
add $0x1, %eax         ; increment register
mov %eax, 0x8049a1c   ; store back to memory

If the OS interrupts Thread 1 between the load and the store, and Thread 2 runs its own full increment, Thread 1 will overwrite Thread 2's result when it resumes. Both threads ran once, but the counter only increased by 1 instead of 2. The result is indeterminate.

Race Condition

Result depends on the timing/interleaving of thread execution. Outcome is indeterminate.

Critical Section

A piece of code that accesses a shared variable/resource and must not be executed by more than one thread simultaneously.

Mutual Exclusion

The guarantee that only one thread executes the critical section at a time — the fundamental property locks provide.

The wish for atomicity: Ideally we would have a single hardware instruction like memory-add 0x8049a1c, $0x1 that performs load-increment-store as one indivisible unit. In general such instructions do not exist, so we use locks to simulate atomicity over multi-instruction sequences.

3. Thread API (POSIX pthreads)

Creating Threads — pthread_create

int pthread_create(
    pthread_t            *thread,
    const pthread_attr_t *attr,
    void                 *(*start_routine)(void *),
    void                 *arg
);
  • thread — handle used to refer to the new thread later
  • attr — attributes (stack size, scheduling priority); pass NULL for defaults
  • start_routine — function the thread begins executing
  • arg — single void * argument passed to the function; cast to any type needed

To pass multiple values, pack them into a struct and pass a pointer to it:

typedef struct __myarg_t {
    int a;
    int b;
} myarg_t;

void *mythread(void *arg) {
    myarg_t *m = (myarg_t *) arg;
    printf("%d %d\n", m->a, m->b);
    return NULL;
}

Waiting for Threads — pthread_join

int pthread_join(pthread_t thread, void **value_ptr);
  • thread — which thread to wait for
  • value_ptr — pointer to receive the thread's return value (pass NULL to ignore)

The caller blocks until the specified thread finishes. The return value is a void *; if you need to return a struct, malloc it inside the thread (do not return a pointer to a local stack variable — the stack is gone after the thread exits).

Locks — pthread_mutex

pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;
pthread_mutex_lock(&lock);
x = x + 1;  // critical section
pthread_mutex_unlock(&lock);

If no thread holds the lock, pthread_mutex_lock acquires it and returns immediately. If another thread holds it, the caller blocks until the lock is released. Two additional variants exist:

  • pthread_mutex_trylock — returns failure immediately if already held (non-blocking)
  • pthread_mutex_timelock — blocks up to a specified timeout

Condition Variables

pthread_cond_t  init = PTHREAD_COND_INITIALIZER;
pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;

// Waiting thread
pthread_mutex_lock(&lock);
while (initialized == 0)
    pthread_cond_wait(&init, &lock);  // releases lock; re-acquires on wake
pthread_mutex_unlock(&lock);

// Signalling thread
pthread_mutex_lock(&lock);
initialized = 1;
pthread_cond_signal(&init);
pthread_mutex_unlock(&lock);
pthread_cond_wait atomically releases the mutex and puts the thread to sleep. When woken by a signal, it re-acquires the mutex before returning. Always check the condition in a while loop (not if) to handle spurious wakeups.

4. Lock Internals — Building a Lock

Building a correct, efficient lock requires cooperation between software, hardware, and the OS. Any implementation is evaluated against three criteria:

Mutual Exclusion

Does the lock prevent multiple threads from being in the critical section simultaneously?

Fairness

Does every contending thread eventually get the lock, or can some threads starve?

Performance

How much time overhead does using the lock add, especially under contention with many threads?

Approach 1 — Disable Interrupts

void lock()   { DisableInterrupts(); }
void unlock() { EnableInterrupts();  }

Earliest approach, designed for single-processor systems. Problems: requires too much trust (a malicious thread never calls unlock and monopolizes the CPU); does not work on multiprocessors; interrupt mask/unmask is slow on modern CPUs.

Approach 2 — Simple Flag (Broken)

void lock(lock_t *mutex) {
    while (mutex->flag == 1)  // TEST
        ;                      // spin-wait
    mutex->flag = 1;           // SET
}

Fails mutual exclusion: two threads can both observe flag == 0 before either sets it to 1. Also wastes CPU with spin-waiting.

Approach 3 — Test-And-Set (Spin Lock)

int TestAndSet(int *ptr, int new) {
    int old = *ptr;
    *ptr = new;
    return old;   // atomic read-modify-write
}

void lock(lock_t *lock) {
    while (TestAndSet(&lock->flag, 1) == 1)
        ;  // spin
}
void unlock(lock_t *lock) { lock->flag = 0; }

TestAndSet is a hardware atomic instruction. It returns the old value and sets the new value in a single indivisible step. Correct (mutual exclusion). Not fair (can starve). Wastes CPU while spinning.

Approach 4 — Compare-And-Swap

int CompareAndSwap(int *ptr, int expected, int new) {
    int actual = *ptr;
    if (actual == expected)
        *ptr = new;
    return actual;
}
void lock(lock_t *lock) {
    while (CompareAndSwap(&lock->flag, 0, 1) == 1)
        ;  // spin
}

CAS updates the value only if it currently equals the expected value, then returns the actual old value. More powerful than TAS — it forms the basis of lock-free data structures. For a simple spin lock, behavior is identical to TestAndSet.

Approach 5 — Load-Linked / Store-Conditional (LL/SC)

void lock(lock_t *lock) {
    while (1) {
        while (LoadLinked(&lock->flag) == 1)
            ;  // spin until zero
        if (StoreConditional(&lock->flag, 1) == 1)
            return;  // success
        // else: retry
    }
}

LL loads a value and marks the address. SC succeeds only if no other store to that address has occurred since the LL. Used on MIPS, ARM, and RISC-V architectures.

Approach 6 — Fetch-And-Add / Ticket Lock

int FetchAndAdd(int *ptr) {
    int old = *ptr;
    *ptr = old + 1;
    return old;
}
void lock(lock_t *lock) {
    int myturn = FetchAndAdd(&lock->ticket);
    while (lock->turn != myturn)
        ;  // spin
}
void unlock(lock_t *lock) {
    FetchAndAdd(&lock->turn);
}
Ticket Lock advantage: Every thread that calls lock() gets a unique ticket number. The lock advances turn one step at a time, ensuring every thread eventually gets the lock — providing fairness (no starvation).

5. Avoiding Spinning — OS-Assisted Locks

All spin-based approaches waste CPU cycles when a thread must wait. The solution is OS support to put waiting threads to sleep.

Just Yield

void lock() {
    while (TestAndSet(&flag, 1) == 1)
        yield();  // give up the CPU
}

yield() moves the calling thread from running to ready. This avoids burning CPU on a full time slice but still has overhead from context switching and does not prevent starvation.

Queue-Based Sleep Lock (park/unpark)

The most robust approach: a queue tracks waiting threads; threads go to sleep (park()) instead of spinning, and are woken explicitly (unpark(threadID)) when the lock is released.

void lock(lock_t *m) {
    while (TestAndSet(&m->guard, 1) == 1)
        ;  // spin on guard only (brief)
    if (m->flag == 0) {
        m->flag = 1;   // acquire lock
        m->guard = 0;
    } else {
        queue_add(m->q, gettid());
        setpark();     // indicate we are about to sleep
        m->guard = 0;
        park();        // go to sleep
    }
}
void unlock(lock_t *m) {
    while (TestAndSet(&m->guard, 1) == 1)
        ;
    if (queue_empty(m->q))
        m->flag = 0;
    else
        unpark(queue_remove(m->q));  // hand lock to next waiter
    m->guard = 0;
}
Wakeup/waiting race: If the lock is released just before a thread calls park(), the thread sleeps forever. Solaris solves this with setpark() — if unpark arrives before park is called, park returns immediately without sleeping.

Two-Phase Locks

A pragmatic hybrid: in the first phase the lock spins for a short time, hoping the holder will release it soon. If the lock is still not acquired, the thread enters the second phase and goes to sleep. This amortizes the context-switch cost when locks are typically held briefly.
