Topic 5: TLB, Advanced Page Tables & Swapping Mechanisms and Policies

Why Paging is Slow — The TLB Motivation

In basic paging every memory access requires at least two physical memory operations: one to look up the page table entry (PTE) and one to fetch the actual data. With a 32-bit address space and 4 KB pages the page table alone consumes 4 MB per process. Multiply that by 100 running processes and you have 400 MB just for bookkeeping.

The solution to the speed problem is the Translation Lookaside Buffer (TLB) — a small, extremely fast hardware cache built into the Memory Management Unit (MMU) that stores the most recently used virtual-to-physical address translations.

Core idea: Instead of going to the page table in RAM on every access, check the TLB first. If the translation is already there (a TLB hit), skip the page table entirely. A typical TLB has 32–128 entries and is searched in parallel using full-associative matching.

TLB Algorithm

On every memory reference the hardware extracts the VPN from the virtual address and searches the TLB:

1: VPN  = (VirtualAddress & VPN_MASK) >> SHIFT
2: (Success, TlbEntry) = TLB_Lookup(VPN)
3: if (Success == TRUE) {          // TLB Hit
4:   if (CanAccess(TlbEntry.ProtectBits))
5:     PhysAddr = (TlbEntry.PFN << SHIFT) | (VirtualAddress & OFFSET_MASK)
6:     AccessMemory(PhysAddr)
7:   else RaiseException(PROTECTION_FAULT)
8: } else {                        // TLB Miss
9:   PTEAddr = PTBR + (VPN * sizeof(PTE))
10:  PTE = AccessMemory(PTEAddr)
11:  if (!PTE.Valid) RaiseException(SEGFAULT)
12:  TLB_Insert(VPN, PTE.PFN, PTE.ProtectBits)
13:  RetryInstruction()
14: }

Spatial and Temporal Locality

Temporal Locality
  • A recently accessed page will likely be accessed again soon
  • Keeping the same translation in the TLB pays off across repeated accesses
Spatial Locality
  • If address x is accessed, nearby addresses are likely accessed soon
  • Array traversal: elements on the same page only cause 1 TLB miss for the whole page

Example: accessing a 10-element array where elements share pages produces only 3 TLB misses and 7 hits (70 % hit rate) because elements on the same physical page reuse the same TLB entry.

Who Handles a TLB Miss — CISC vs. RISC

CISC — Hardware-Managed TLB (e.g. x86 Intel)
  • Hardware walks the page table automatically on a miss
  • Hardware knows the page-table base register (CR3 on x86)
  • Finds the correct PTE, updates the TLB, retries the instruction
  • OS is never involved in the miss path
RISC — Software-Managed TLB (e.g. ARM, MIPS)
  • On a TLB miss the hardware raises a trap to the OS
  • OS trap handler looks up the PTE in the page table
  • OS inserts the translation into the TLB and returns from trap
  • Instruction is retried and now finds a TLB hit

TLB Entry Fields

Each TLB entry stores more than just VPN → PFN. Additional bits include:

  • Valid bit — is this entry in use?
  • Protection bits — read / write / execute permissions
  • ASID (Address Space Identifier) — which process owns this entry
  • Dirty bit — has the page been modified?

Context Switching and ASID

When the OS switches processes, VPN 10 of process A maps to PFN 100, while VPN 10 of process B maps to PFN 170. Without distinguishing them, the TLB would serve wrong translations. Two solutions exist:

Flush on Switch

Set all TLB entries invalid on each context switch. Simple but costly — every process restart is a cold cache.

ASID Field

Tag each TLB entry with an address-space identifier. Entries from different processes coexist safely; the hardware matches on both VPN and ASID.

Sharing pages with ASID: Two processes can map different VPNs to the same physical page (e.g., a shared library). Process 1 maps VPN 10 → PFN 101 (ASID 1) and Process 2 maps VPN 50 → PFN 101 (ASID 2). Both entries coexist in the TLB and reduce overall physical memory usage.

TLB Replacement — LRU

When all TLB entries are occupied and a new translation must be inserted, the LRU (Least Recently Used) policy evicts the entry that has not been used for the longest time, exploiting temporal locality.

Advanced Page Tables

The size problem: a 32-bit address space with 4 KB pages and 4-byte PTEs requires a page table of 4 MB per process. Four approaches reduce this overhead:

1. Larger Pages

16 KB pages → 1 MB page table. Fewer entries needed but causes internal fragmentation — a 65 KB allocation wastes almost an entire page.

2. Hybrid (Paging + Segmentation)

One page table per segment (code, heap, stack). Base register points to the segment's page table; bound register limits its size. Avoids storing invalid entries between heap and stack.

3. Multi-level Page Tables

Chop the page table into page-sized pieces. A page directory tracks which pieces exist. Invalid ranges need no physical memory at all.

Multi-level Page Tables in Detail

The virtual address is split into multiple parts: a page directory index, one or more page table indices, and the offset.

Virtual Address (32-bit, 4KB pages, 2-level):
[ PD Index (10 bits) | PT Index (10 bits) | Offset (12 bits) ]

Step 1: Use PD Index to find PDE in Page Directory
Step 2: PDE gives PFN of the Page Table page
Step 3: Use PT Index to find PTE in that Page Table page
Step 4: PTE gives PFN of the actual data page
Step 5: PhysAddr = PFN << 12 | Offset
Advantage: Only allocates page-table space proportional to the address space actually in use. For a process using only code + heap + stack pages (3 clusters), a two-level table may need only 3 page-table pages instead of 16.
Disadvantage: A TLB miss now requires two memory accesses (one for the page directory, one for the page table) instead of one — a time-space trade-off.

For very large address spaces (e.g., x86-64) three or four levels are used. Each additional level adds one memory lookup on a TLB miss but keeps the directory structures manageable in size.

Swapping: Mechanisms

When physical memory is full, the OS must move pages to swap space on disk. Disk is orders of magnitude slower than RAM, so swapping is a last resort — but it allows processes whose combined address spaces exceed physical RAM to coexist.

Swap Space and the Present Bit

Each PTE gains a present bit:

Present bit = 1

Page is in physical memory. Normal TLB/page-table lookup proceeds.

Present bit = 0

Page has been swapped to disk. Accessing it triggers a page fault. The OS must load the page back from disk before the instruction can complete.

The OS keeps track of where on disk each swapped page lives (its swap address) so it can load it back when needed.

When to Swap — Watermarks

Rather than waiting until memory is completely full (the naive "lazy" approach), the OS uses a background page daemon (swap daemon) with two thresholds:

  • Low Watermark (LW): If free pages fall below this threshold, the daemon wakes up and starts evicting pages.
  • High Watermark (HW): The daemon keeps evicting until free pages reach this level, then sleeps again.
Example: LW = 10, HW = 30. When free pages drop to 9 the daemon starts evicting. It stops only when 30 free pages are available again. This keeps a buffer of free pages ready for sudden demand.

Swapping: Replacement Policies

The goal of any replacement policy is to minimize cache misses (page faults). The average memory access time is:

AMAT = P_Hit * T_Memory + P_Miss * T_Disk

Because disk access is ~100,000× slower than RAM, even a small increase in miss rate has a devastating effect on AMAT.

The Optimal Policy (OPT / MIN)

Replace the page that will not be used for the longest time in the future. This is theoretically optimal but not implementable (requires knowledge of future accesses). It serves as the gold-standard benchmark.

FIFO — First In, First Out

Pages are kept in a queue. When a replacement is needed, the oldest page (the one that entered memory first) is evicted, regardless of how often it has been used recently.

Belady's Anomaly: FIFO can produce more page faults when given more physical frames — a counterintuitive result that does not affect stack-based policies like LRU.

Random

Pick a random page to evict. No bookkeeping overhead. Performance varies widely — sometimes as good as OPT, sometimes much worse. Immune to Belady's anomaly and avoids worst-case patterns that defeat LRU.

LRU — Least Recently Used

Evict the page that has not been accessed for the longest time. Exploits temporal locality: recently used pages are likely to be used again.

No-Locality Workload

Random page access pattern. All policies (OPT, LRU, FIFO, Random) perform similarly. Cache size is the dominant factor.

80-20 Workload

80 % of references hit 20 % of pages. LRU outperforms FIFO and Random because it holds on to the "hot" pages.

Looping-Sequential

Access pages 0→49 repeatedly. LRU performs badly — it always evicts the page that is about to be reused. FIFO and Random do better here.

Clock Algorithm — Approximating LRU

True LRU requires hardware timestamps on every page access — expensive. The Clock algorithm approximates LRU using a single use bit per page:

  • Hardware sets the use bit to 1 whenever a page is referenced.
  • A clock hand sweeps circularly through all pages.
  • If the hand finds a page with use bit = 1, it clears the bit and advances.
  • If the hand finds a page with use bit = 0, that page is evicted and replaced.

The clock algorithm performs between pure LRU and FIFO/Random — close to LRU on locality workloads and far better than no-history policies.

Dirty Pages, Prefetching and Clustering

Dirty Bit

If a page has been modified (dirty bit = 1), evicting it requires writing it back to disk — costly. Clean pages can be evicted "for free." Policies prefer evicting clean pages when possible.

Prefetching

The OS predicts which pages will be needed soon (e.g., page N+1 after page N is loaded) and fetches them proactively, hiding disk latency.

Clustering (Grouping): Rather than writing one dirty page to disk at a time, the OS collects a batch of pending writes and issues a single large disk write. One large I/O is far more efficient than many small ones.

Thrashing

When the combined working sets of all running processes exceed available physical memory, the system spends more time swapping pages in and out than doing real work. This is called thrashing.

Solution: The OS can detect thrashing and reduce the degree of multiprogramming — suspending some processes so the remaining ones' working sets fit in memory. CPU utilization drops sharply at the thrashing point.
